01 · Background

What is the Payment Module?

The Payment Module is TFW's core financial transaction engine — used across Move In, Move Out, Reservation, Retail Sale, and standalone payment flows. It handles five payment method types, configurable surcharge and discount modifiers, prepayment with force rent assessment, bill-ahead logic, tenant alert integration, daily late event processing at the time of payment, bulk batch payments for cash/check/money order and a separate autopay credit card batch run — all underpinned by a rich configuration layer and granular role-based permissions.

02 · Single Payment Workflow

How a payment is processed

Every payment flow — whether triggered from Move In, Move Out, Ledger Review, or Quick Link — follows the same structured sequence, with the payment screen adapting based on configuration, tenant state, and permissions.

Step 01 · Authentication

Credential Gate

  • If "Require Credentials" is configured, user re-authenticates before the payment screen loads
  • Tenant alerts surface immediately — "Do Not Rent" or "Do Not Accept Checks" shown with log of ignored alerts
  • Daily late event processing can be triggered here before payment proceeds
Step 02 · Payment Details

Capture Payment

  • Tenant and ledger confirmed, pay amount set (validated if over configured threshold)
  • Effective date defaults to today — editable by users with permission
  • Payment method selected — defaulting to ledger's saved method or Cash
  • Surcharge / discount shown if configured; waivable by permitted users
  • Check / MO number captured; autopay sign-up available for CC and ACH
  • "Do Not Authorize" option available for credit card — records payment without processing card
Step 03 · Bill Ahead (Optional)

Advance Payment

  • User enters a future date — system shows all bills that will be generated up to that date
  • User can choose to generate the bills and collect upfront, or take an advance payment without generating bills
  • Facilities without prepayment allowed must always generate bills first
Step 04 · Post Payment

Outcome & Delivery

  • Success / failure notification with credit card approval or response code shown
  • Receipt generated and delivered per configured delivery method (always printed for cash)
  • Partial payment letter generated if payment was partial and config is enabled
  • Credit card decline letter generated on decline if configured
  • Saved payment method and autopay state updated on ledger
  • Force rent assessment triggered if prepayment threshold exceeded
03 · Payment Methods

Five payment method types

Each payment method is independently configured in the Financial → Payment Method settings — with active/inactive status, modifier (surcharge/discount) logic, export descriptions, and payment number behaviour.

💵

Cash

Default payment method. No additional fields required. Receipts are always printed for cash payments regardless of delivery configuration.

Always Print Receipt
💳

Credit Card

Supports card number, expiry, CVV, name, and billing address. Can be tokenised and saved for autopay. Supports surcharge/discount modifier and "Do Not Authorize" recording mode.

Card Number Expiry CVV Billing Address Autopay Tokenised
🏦

ACH

Bank transfer method capturing account number, routing number, bank name, and account type (checking/savings). Saved on ledger for autopay if payment succeeds.

Account No. Routing No. Bank Name Acct Type Autopay
📄

Check

Configurable to show and/or mandate check number input. Integrates with the "Refuse Check from flagged tenants" configuration to block or warn on checks from tenants with active alerts.

Check Number Mandatory Toggle
📮

Money Order

Money order number is mandatory when this method is selected (added in v2 of Payment Method config). Supports the same Show / Mandate payment number configuration pattern as checks.

MO Number (Required)
⚙️

Surcharge / Discount Modifier

Each method can have a modifier configured — Surcharge (forward card processor fees to tenant) or Discount (give discount for cash to offset inflated rates). Calculation styles: Flat, Percent, or Higher of Flat or Percent. Waivable per payment by permitted users.

Flat Percent Higher of Both Charge Category
04 · Iterative Delivery

Versioned payment evolution

The payment module was delivered in multiple versioned iterations — each addressing real user pain points, client requests, or migration requirements.

💳
Core Payment Flow 3 versions + 1 patch
v1.0Foundation

Full payment integration across all workflows

Built payment processing across Move In, Move Out, Reservation, and Retail Sale. Implemented all 5 payment methods, credential gate, tenant alert integration, daily late event billing at payment time, bill-ahead for advance payments, surcharge/discount display, save-for-future and autopay sign-up, and receipt/letter delivery.

v2.0Website Restrictions

Locked down tenant-facing website payment behaviour

Restricted the tenant-facing website from allowing: effective date modification, Cash/Check/ACH payment types, credential re-entry, Bill Ahead, promo plan application, and refund initiation — ensuring the online portal only supports credit card payments on today's date.

v3.0Next Amount Fix

Removed auto-fill of next payable amount at zero balance

Fixed a UX problem where the payment screen automatically pre-filled the next period's payable amount when the ledger reached zero — causing accidental overpayments. The next payable amount is now only populated on explicit user request. Requested by 9G North Storage (FID: 29684).

v3.1Autofill Patch

Blank field on amount clear — no auto-refill

Fixed a related issue (reported by Catskill's Mikey) where clearing the payment amount field caused it to immediately re-populate with the balance. Amount field now stays blank when cleared, and the Pay button is disabled until a valid amount is entered.

⚙️
Financial · Payment Configuration 3 versions
v1.0Foundation

Core payment configuration with prepayment strategies

Built the full payment configuration panel — credential requirement, mandate comment, check refusal from flagged tenants, payment validation threshold, and three prepayment strategies: Advance Paid To Date (deprecated), Force Rent Assessment, and Allow Partial Credit (forces bills only for fully-coverable periods, holding remainder as prepaid credit).

v2.0Partial Payments

Configurable partial payment restriction for delinquent tenants

Added a facility-level toggle to prevent partial payments from delinquent tenants — blocking the payment amount field and showing an explanatory message when a delinquent tenant attempts to pay less than the full balance. Applies on both the staff interface and the tenant-facing website (takes highest priority over website config).

v3.0Tax Numbers

GST / PST / HST tax numbers on receipts

Added support for Canadian tax number fields (GST, PST, HST) in payment configuration — once entered, these are reflected on all new receipts and all previously migrated receipt documents. Migrated values from v10 during platform upgrade.

Full payment method setup with modifier engine

Built the payment method configuration — description, active/inactive status, 6 type options (Credit Card, ACH, Check, Cash, Money Order, Other), alternate export description for QuickBooks, show/mandate payment number toggle, and the full modifier engine (Surcharge/Discount) with flat, percent, or higher-of-both calculation styles and linked charge category.

05 · Batch Payment

Two modes of bulk payment processing

The Batch Payment module lets facility staff process payments for multiple tenants at once — with two distinct modes covering cash/check/money order collection and automated credit card autopay runs.

Mode 01 · Cash / Check / Money Order

Manual Batch — Bulk Ledger Update

Designed for end-of-day collection — staff select tenants, enter amounts, and post all payments at once. Supports editable amounts, per-tenant receipt and invoice generation, pay-off-balance shortcuts, bulk payment type updates, and effective date override.

  • Filter by tenants with balance due or all tenants
  • Editable amount per tenant
  • Bulk "Pay Off Balances" sets all amounts to current balance
  • Bulk "Reset" clears all entered amounts
  • Per-tenant receipt (default on) and invoice (default off) toggle
  • Effective date defaulted to today, editable per-tenant or for entire batch
  • Check number / money order number captured per tenant
  • Scan check support (future — hardware integration)
  • Payment lock — other users cannot access payments mid-batch
  • v2 added: Cash, Check, Money Order method selection per tenant
Mode 02 · Autopay Credit Card Batch

CC Autopay Run — Bulk Card Processing

Designed for facilities that manually trigger credit card autopay runs rather than relying on automated scheduling. Processes all tenants with tokenised cards and autopay enabled in a single batch.

  • Filter: Autopay-enabled tenants (default) or tenants with balance + saved card
  • Only shows tenants with a stored card token — no token, no row
  • Shows tenant balance, payment amount, surcharge/discount per method
  • Approval / decline code shown per tenant after processing
  • Per-tenant receipt generation (default off, select/unselect all)
  • Credit card decline letter generated on failure
  • Summary: total cards to run + total amount to authorise shown upfront
  • Print grid option to export the full batch list
  • All ledger balances updated in a single batch commit

07 · Permissions

Role-based payment access

Three granular permissions control who can access the payment screen, modify transaction dates, and process negative-amount corrections — giving facility operators precise control over payment authority.

access_screen

Users can access the payment screen and take payments from tenants. Without this permission, payment screens are inaccessible regardless of workflow context.

modify_effective_date

Users can change the effective date of a payment from today's default — allowing backdating or future-dating transactions for reconciliation purposes.

negative_amount

Users can enter a negative payment amount — used for manual ledger corrections and reversal entries where a formal refund flow isn't being used.

08 · Outcomes

Key outcomes & contributions

The Payment Module touches every tenant financial interaction in the platform — demanding accuracy across billing logic, security, multi-method processing, and batch performance.

01

Full 5-method payment engine integrated across all platform workflows

Built a unified payment screen consumed by Move In, Move Out, Ledger Review, Retail Sale, and standalone payment — supporting Cash, Credit Card, ACH, Check, and Money Order with method-specific field capture, tokenisation, autopay sign-up, and surcharge/discount calculation per method type.

02

Configurable prepayment engine with three distinct billing strategies

Designed and implemented three prepayment strategies — Advance Paid To Date (deprecated), Force Rent Assessment (bills future periods to match overpayment), and Allow Partial Credit (holds partial-period overpayment as a credit) — with detailed worked examples covering edge cases like multi-unit tenants and varying payment amounts.

03

Two-mode batch payment system for high-volume facility operations

Built two independently scoped batch payment modes — a manual cash/check/MO batch for end-of-day collection with editable amounts and bulk shortcuts, and a CC autopay batch for bulk card processing with token validation, approval/decline codes, and per-tenant decline letters — with a payment lock preventing concurrent access mid-batch.

04

Surcharge and discount modifier engine with facility-level configuration

Implemented a per-payment-method modifier engine supporting Flat, Percent, and Higher-of-both calculation styles — enabling facilities to forward card processor fees to tenants as surcharges, or implement cash-discount pricing strategies where state law prohibits surcharges on cards.

05

Partial payment control and delinquency enforcement across staff and web portals

Built a configurable partial payment restriction that locks payment amounts for delinquent tenants on both the staff interface and the tenant-facing website — with the staff setting taking higher priority over any website configuration, preventing bypasses through the online portal.

06

UX fixes resolving real-world accidental overpayment patterns

Resolved two related payment UX issues — the auto-fill of next payable amount at zero balance (causing accidental double payments, reported by 9G North Storage) and the auto-refill when the amount field was cleared (reported by Catskill) — improving payment accuracy across the user base.
